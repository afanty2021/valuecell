## 配置优先级

ValueCell项目采用**三层配置系统**，以实现从开发到生产的灵活部署。配置解析遵循以下优先级顺序（从高到低）：

1.  **环境变量** - 运行时覆盖（例如 `OPENROUTER_API_KEY`）
2.  **.env 文件** - 用户级配置（位于项目根目录）
3.  **YAML 文件** - 系统默认值（位于 `python/configs/` 目录）

此分层结构允许您：
- 通过 `.env` 文件设置提供商凭据，而无需修改代码
- 通过环境变量在运行时覆盖设置
- 在YAML文件中维护合理的默认值

## 快速入门

### 第一步：获取API密钥

ValueCell支持多种LLM提供商。至少选择一个：

| 提供商 | 注册链接 |
| :--- | :--- |
| **OpenRouter** | [openrouter.ai](https://openrouter.ai/) |
| **SiliconFlow** | [siliconflow.cn](https://www.siliconflow.cn/) |
| **Google** | [ai.google.dev](https://ai.google.dev/) |
| **OpenAI** | [platform.openai.com](https://platform.openai.com/) |
| **DashScope** | [bailian.console.aliyun.com](https://bailian.console.aliyun.com/#/home) |

### 第二步：配置 .env 文件

复制示例文件并添加您的API密钥：

```bash
# 在项目根目录
cp .env.example .env
```

编辑 `.env` 文件并添加您的凭据：

```bash
# OpenRouter
OPENROUTER_API_KEY=sk-or-v1-xxxxxxxxxxxxx

# 或 SiliconFlow (最适合中文模型和成本)
SILICONFLOW_API_KEY=sk-xxxxxxxxxxxxx

# 或 Google Gemini
GOOGLE_API_KEY=AIzaSyDxxxxxxxxxxxxx

# 可选：设置主要提供商（未设置时自动检测）
PRIMARY_PROVIDER=openrouter
```

### 第三步：启动应用程序

```bash
# macOS / Linux
bash start.sh

# Windows PowerShell
.\start.ps1
```

系统将根据配置的API密钥自动检测可用的提供商。

> **注意**：如果遇到数据库兼容性错误，请删除以下位置：
> - LanceDB 目录（系统应用程序目录，与 `.env` 相同）：
>   - macOS: `~/Library/Application Support/ValueCell/lancedb`
>   - Linux: `~/.config/valuecell/lancedb`
>   - Windows: `%APPDATA%\\ValueCell\\lancedb`
> - Knowledge 目录（系统应用程序目录，与 `.env` 相同）：
>   - macOS: `~/Library/Application Support/ValueCell/.knowledge`
>   - Linux: `~/.config/valuecell/.knowledge`
>   - Windows: `%APPDATA%\\ValueCell\\.knowledge`
> - SQLite 数据库文件（系统应用程序目录，与 `.env` 相同）：
>   - macOS: `~/Library/Application Support/ValueCell/valuecell.db`
>   - Linux: `~/.config/valuecell/valuecell.db`
>   - Windows: `%APPDATA%\\ValueCell\\valuecell.db`

### 配置解析工作原理

#### 1. 提供商配置加载

当系统需要一个模型时，它会执行以下步骤：

1.  **加载提供商YAML** (例如 `configs/providers/openrouter.yaml`)
2.  **解析YAML中的 `${VAR}` 占位符**，使用环境变量
3.  **应用环境变量覆盖** (例如 `OPENROUTER_API_KEY` 覆盖 `connection.api_key`)
4.  **返回包含已解析值的 ProviderConfig 对象**

**示例：OpenRouter 配置**

```yaml
connection:
  base_url: "https://openrouter.ai/api/v1"
  api_key_env: "OPENROUTER_API_KEY"    # 指定要使用的环境变量

default_model: "anthropic/claude-haiku-4.5"

defaults:
  temperature: 0.5
  max_tokens: 4096
```

系统会自动从 `.env` 或环境中读取 `OPENROUTER_API_KEY`。

#### 2. 智能体配置加载

当您创建一个智能体（例如 `research_agent`）时，系统会：

1.  **加载智能体YAML** (例如 `configs/agents/research_agent.yaml`)
2.  **解析 `${VAR}` 占位符**（例如，从环境变量中获取API密钥）
3.  **应用环境变量覆盖**，通过 `env_overrides` 映射
4.  **与全局默认值合并**，来自 `config.yaml`
5.  **返回包含完整配置的 AgentConfig 对象**

**示例：智能体配置**

```yaml
name: "Research Agent"
enabled: true

models:
  primary:
    model_id: "google/gemini-2.5-flash"
    provider: "openrouter"
    provider_models:
      siliconflow: "Qwen/Qwen3-235B-A22B-Thinking-2507"
      google: "gemini-2.5-flash"
    parameters:
      temperature: 0.7

env_overrides:
  RESEARCH_AGENT_MODEL_ID: "models.primary.model_id"
  RESEARCH_AGENT_PROVIDER: "models.primary.provider"
```

这允许运行时覆盖：

```bash
export RESEARCH_AGENT_MODEL_ID="anthropic/claude-3.5-sonnet"
export RESEARCH_AGENT_PROVIDER="openrouter"
# 现在研究智能体使用Claude 3.5 Sonnet而不是Gemini
```

## 详细配置参考

### 全局配置 (`config.yaml`)

主配置文件设置系统范围的默认值：

```yaml
models:
  # 如果多个提供商的API密钥都可用，则使用的主要提供商
  primary_provider: "openrouter"
  
  # 全局默认参数（除非被覆盖，否则所有模型都使用）
  defaults:
    temperature: 0.5
    max_tokens: 4096
  
  # 提供商注册表
  providers:
    openrouter:
      config_file: "providers/openrouter.yaml"
      api_key_env: "OPENROUTER_API_KEY"
    siliconflow:
      config_file: "providers/siliconflow.yaml"
      api_key_env: "SILICONFLOW_API_KEY"
    google:
      config_file: "providers/google.yaml"
      api_key_env: "GOOGLE_API_KEY"
    dashscope:
      config_file: "providers/dashscope.yaml"
      api_key_env: "DASHSCOPE_API_KEY"

# 智能体注册表
agents:
  super_agent:
    config_file: "agents/super_agent.yaml"
  research_agent:
    config_file: "agents/research_agent.yaml"
  auto_trading_agent:
    config_file: "agents/auto_trading_agent.yaml"
```

### 提供商配置

每个提供商在 `configs/providers/` 中都有自己的YAML文件。其结构如下：

```yaml
name: "提供商显示名称"
provider_type: "provider_id"           # 内部使用
enabled: true                          # 可以在不删除配置的情况下禁用

# 连接详情
connection:
  base_url: "https://api.example.com/v1"
  api_key_env: "PROVIDER_API_KEY"      # 要读取的环境变量
  endpoint_env: "PROVIDER_ENDPOINT"    # 可选：用于Azure风格的端点

# 未指定时的默认模型
default_model: "model-id"

# 该提供商所有模型的默认参数
defaults:
  temperature: 0.7
  max_tokens: 4096
  top_p: 0.95

# 可用模型列表
models:
  - id: "model-id-1"
    name: "模型显示名称"
    context_length: 128000
    max_output_tokens: 8192
  
  - id: "model-id-2"
    name: "另一个模型"
    context_length: 256000

# 嵌入配置（可选，不是所有提供商都支持）
embedding:
  default_model: "embedding-model-id"
  
  defaults:
    dimensions: 1536
    encoding_format: "float"
  
  models:
    - id: "embedding-model-id"
      name: "嵌入模型"
      dimensions: 1536
      max_input: 8192

# 提供商特定配置
extra_headers:
  HTTP-Referer: "https://valuecell.ai"
  X-Title: "ValueCell"
```

### 智能体配置

智能体YAML文件定义了如何初始化智能体。关键特性：

```yaml
name: "智能体显示名称"
enabled: true

# 模型配置
models:
  # 主要推理模型
  primary:
    model_id: "model-id"               # 可以使用提供商前缀（例如 "anthropic/claude-3.5-sonnet"）
    provider: "openrouter"             # 必须明确指定（不会自动检测）
    
    # 不同提供商的备用模型
    provider_models:
      siliconflow: "qwen/qwen3-max"
      google: "gemini-2.5-flash"
    
    # 模型特定参数（覆盖提供商默认值）
    parameters:
      temperature: 0.8
      max_tokens: 8192
  
  # 可选：单独的嵌入模型配置
  embedding:
    model_id: "embedding-model-id"
    provider: "siliconflow"
    provider_models:
      google: "gemini-embedding-001"
    parameters:
      dimensions: 2560

# 将环境变量映射到配置路径以进行运行时覆盖
env_overrides:
  # 语法：ENV_VAR -> config.path.to.value
  AGENT_MODEL_ID: "models.primary.model_id"
  AGENT_PROVIDER: "models.primary.provider"
  AGENT_TEMPERATURE: "models.primary.parameters.temperature"
  AGENT_MAX_TOKENS: "models.primary.parameters.max_tokens"
  
  # 嵌入配置
  AGENT_EMBEDDER_MODEL: "models.embedding.model_id"
  AGENT_EMBEDDER_PROVIDER: "models.embedding.provider"
```

## 提供商自动检测与回退

### 自动检测

ValueCell根据可用的API密钥自动选择主要提供商：

**优先顺序**（如果多个提供商都有API密钥）：
1.  OpenRouter
2.  SiliconFlow
3.  Google
4.  OpenAI
5.  OpenAI-Compatible
6.  Azure
7.  其他已配置的提供商（包括DashScope、DeepSeek等）

通过环境变量覆盖此设置：

```bash
export PRIMARY_PROVIDER=siliconflow
```

或禁用自动检测：

```bash
export AUTO_DETECT_PROVIDER=false
```

### 回退机制

如果主要提供商失败，ValueCell会自动尝试备用提供商。

**回退链**（从已启用的提供商中自动填充）：
- 除主要提供商外，所有具有有效API密钥的提供商
- 在第一个成功创建模型时停止

要覆盖备用提供商：

```bash
export FALLBACK_PROVIDERS=siliconflow,google
```

要禁用回退：

```yaml
# 在智能体YAML中
use_fallback: false
```

### 提供商特定模型映射

使用回退时，智能体可以指定每个提供商要使用的模型：

```yaml
# 在智能体配置中
models:
  primary:
    model_id: "anthropic/claude-haiku-4.5"
    provider: "openrouter"
    
    # 如果OpenRouter失败，则为备用提供商使用这些模型
    provider_models:
      siliconflow: "zai-org/GLM-4.6"      # 类似能力
      google: "gemini-2.5-flash"          # 快速高效
```

当发生回退时：
1.  尝试使用 `anthropic/claude-haiku-4.5` 的OpenRouter
2.  如果失败，尝试使用 `zai-org/GLM-4.6` 的SiliconFlow
3.  如果失败，尝试使用 `gemini-2.5-flash` 的Google

## 配置模式

### 模式一：带回退的多模型设置

**用例**：高可用性与成本优化

```bash
# .env 文件
OPENROUTER_API_KEY=sk-or-v1-xxxxx        # 主要：访问多种模型
SILICONFLOW_API_KEY=sk-xxxxx             # 备用：成本效益高
GOOGLE_API_KEY=AIzaSyD-xxxxx             # 第二备用：专业
DASHSCOPE_API_KEY=sk-xxxxx               # DashScope：Qwen3模型（中文优化）

# config.yaml
models:
  primary_provider: "openrouter"          # 主要（最佳模型）
  # 备用自动填充为 [siliconflow, google]
```

### 模式二：为智能体定制模型

**用例**：为每个任务优化智能体

```yaml
# 在 research_agent.yaml 中
models:
  primary:
    provider: "openrouter"
    model_id: "anthropic/claude-3.5-sonnet"  # 最适合研究
    
  embedding:
    provider: "siliconflow"
    model_id: "Qwen/Qwen3-Embedding-4B"      # 最佳嵌入
```

### 模式三：开发与生产环境

```bash
# .env.production  
OPENROUTER_API_KEY=sk-or-v1-prod-xxxxx
SILICONFLOW_API_KEY=sk-prod-xxxxx
APP_ENVIRONMENT=production
```

然后创建 `config.production.yaml` 并配置生产环境特定的设置。

### 模式四：运行时覆盖

**用例**：在不更改代码的情况下对不同模型进行A/B测试

```bash
# 测试不同模型的脚本
for model in "gpt-4o" "claude-3.5-sonnet" "gemini-2.5-flash"; do
    echo "Testing: $model"
    RESEARCH_AGENT_MODEL_ID="$model" python your_script.py
done
```

## 开发者指南

### 配置系统架构

配置系统分为三层：

1.  **加载器层** (`valuecell/config/loader.py`)
    - 读取YAML文件
    - 解析 `${VAR}` 占位符
    - 应用环境变量覆盖
    - 实现缓存

2.  **管理器层** (`valuecell/config/manager.py`)
    - 高级配置访问
    - 提供商验证
    - 模型工厂集成
    - 回退链管理

3.  **工厂层** (`valuecell/adapters/models/factory.py`)
    - 创建实际的模型实例
    - 提供商特定的实现
    - 参数合并
    - 错误处理和回退

### 创建模型

```python
from valuecell.utils.model import get_model, get_model_for_agent

# 使用默认配置
model = get_model("PLANNER_MODEL_ID")

# 用kwargs覆盖
model = get_model("RESEARCH_AGENT_MODEL_ID", temperature=0.9, max_tokens=16384)

# 获取智能体特定的模型
model = get_model_for_agent("research_agent", temperature=0.8)

# 使用特定提供商
from valuecell.utils.model import create_model_with_provider
model = create_model_with_provider("openrouter", "anthropic/claude-3.5-sonnet")
```

### 添加新提供商

1.  **创建提供商YAML** (`configs/providers/my_provider.yaml`)
2.  **在 `valuecell/adapters/models/factory.py` 中实现提供商类**
3.  **在 `ModelFactory._providers` 中注册提供商**
4.  **在 `config.yaml` 提供商注册表中添加**
5.  **为提供商配置添加测试**

## 最佳实践

1.  **在 .env 中设置API密钥**
    - 切勿将API密钥提交到版本控制中
    - 使用 `.gitignore` 排除 `.env`
    - 在CI/CD中使用环境变量

2.  **使用提供商回退**
    - 为可靠性配置多个提供商
    - 在智能体中指定 `provider_models` 以实现一致的回退
    - 在部署前测试回退行为

3.  **监控配置**
    - 记录配置选择决策
    - 在启动时验证配置
    - 在生产环境中缺少API密钥时发出警报

4.  **版本化您的配置**
    - 将智能体配置保留在版本控制中
    - 记录为什么选择特定模型
    - 在代码审查中审查配置更改

5.  **优化成本**
    - 为简单任务使用更便宜的模型
    - 为实时应用程序使用更快的模型
    - 监控API使用情况并设置支出限制

## 故障排除

- **数据库兼容性问题**：如果遇到数据库错误，请删除系统应用目录中的 `lancedb`、`.knowledge` 和 `valuecell.db` 文件。
- **缺少API密钥**：确保在 `.env` 文件中设置了所有必需的API密钥，并且环境变量名称与YAML配置中的 `api_key_env` 字段匹配。
- **模型创建失败**：检查提供商是否已启用，API密钥是否正确，并且模型ID在提供商的YAML文件中列出。
- **回退未按预期工作**：验证 `FALLBACK_PROVIDERS` 环境变量是否正确设置，或检查 `fallback_providers` 属性以查看自动检测的回退链。

## 国际化支持

ValueCell系统支持多语言和区域设置，以提供本地化的用户体验。系统支持以下语言：

- **英语 (en)**
- **简体中文 (zh_CN)**
- **繁体中文 (zh_TW)**
- **日语 (ja)**

### 语言检测与区域设置

系统通过以下方式自动检测用户的语言和区域：

1.  **浏览器语言检测**：通过HTTP `Accept-Language` 头部检测用户首选语言。
2.  **IP地理位置检测**：通过用户IP地址确定用户所在区域（如中国内地或其它地区）。

**示例：浏览器语言检测**
```python
from valuecell.utils.i18n_utils import detect_browser_language

# 从HTTP请求头获取
accept_language = "zh-CN,en;q=0.9"
detected_lang = detect_browser_language(accept_language)
# 返回 "zh_CN"
```

### 时区与格式化

系统根据用户语言自动设置默认时区和格式化规则：

| 语言 | 默认时区 | 日期格式 | 货币符号 |
| :--- | :--- | :--- | :--- |
| **en** | America/New_York | %m/%d/%Y | $ |
| **zh_CN** | Asia/Shanghai | %Y年%m月%d日 | ¥ |
| **zh_TW** | Asia/Hong_Kong | %Y年%m月%d日 | HK$ |
| **ja** | Asia/Tokyo | %Y/%m/%d | ¥ |

### 国际化API

系统提供RESTful API来管理国际化设置：

- `GET /i18n/config` - 获取当前国际化配置
- `GET /i18n/languages` - 获取支持的语言列表
- `PUT /i18n/language` - 设置用户语言
- `PUT /i18n/timezone` - 设置用户时区
- `POST /i18n/detect-language` - 检测浏览器语言

**示例：设置用户语言**
```bash
curl -X PUT http://localhost:8000/i18n/language \
  -H "Content-Type: application/json" \
  -H "X-User-ID: user123" \
  -d '{"language": "zh_CN"}'
```

## 模型可用性检查

系统提供了模型可用性检查功能，允许用户在前端界面中验证API密钥的有效性和模型的可用性。

### 检查机制

1.  **API密钥验证**：系统通过调用提供商的API端点来验证API密钥的有效性。
2.  **模型可用性检查**：系统检查指定模型ID是否在提供商的可用模型列表中。

### 前端实现

在设置界面中，用户可以点击"检查可用性"按钮来验证配置：

```json
// 前端翻译文件 (zh_CN.json)
{
  "settings": {
    "models": {
      "checkAvailability": "检查可用性",
      "available": "可用",
      "unavailable": "不可用"
    }
  }
}
```

### 后端验证

后端通过 `ConfigManager` 类的 `validate_provider` 方法执行验证：

```python
from valuecell.config.manager import get_config_manager

manager = get_config_manager()
is_valid, error_msg = manager.validate_provider("openrouter")
if not is_valid:
    print(f"验证失败: {error_msg}")
```

此功能确保用户在配置模型提供商时能够及时发现并修复配置错误。
